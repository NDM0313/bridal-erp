# Final Owner Rules (Do-Not-Break)

**Purpose**: Keep ERP clean, accurate, and scalable long-term  
**Goal**: ERP remains reliable and trusted for years

---

## What Owner Must NEVER Allow

### 1. ❌ NEVER Allow Old System Usage

**Rule**: Once ERP is live, old system must NEVER be used for daily operations.

**Why**:
- Creates duplicate data
- Breaks single source of truth
- Confuses team
- Wastes time

**Enforcement**:
- Stop old system immediately on go-live
- Remove access if possible
- Make ERP mandatory
- Check daily if old system is being used

**Action if Violated**:
- Remind user ERP is mandatory
- Remove access to old system
- Provide support for ERP usage
- Document violation

---

### 2. ❌ NEVER Allow Data Entry Outside ERP

**Rule**: All business data must be entered in ERP, nothing outside.

**Why**:
- ERP is single source of truth
- Outside data creates confusion
- Cannot make decisions from incomplete data
- Wastes time and effort

**Enforcement**:
- Make ERP data entry mandatory
- Check daily if data is being entered
- Verify all data is in ERP
- Don't accept data from outside sources

**Action if Violated**:
- Remind user to enter data in ERP
- Provide training if needed
- Verify data is entered correctly
- Document violation

---

### 3. ❌ NEVER Allow Skipping Procedures

**Rule**: All users must follow daily checklists and procedures.

**Why**:
- Procedures ensure data accuracy
- Procedures prevent mistakes
- Procedures maintain system integrity
- Skipping procedures breaks system

**Enforcement**:
- Reinforce procedures daily
- Check compliance regularly
- Explain why procedures matter
- Celebrate compliance

**Action if Violated**:
- Remind user of procedures
- Provide training if needed
- Check why procedure was skipped
- Document violation

---

### 4. ❌ NEVER Allow Ignoring Errors

**Rule**: All errors must be reported and fixed immediately.

**Why**:
- Errors indicate problems
- Ignored errors get worse
- Errors break trust in system
- Errors affect data accuracy

**Enforcement**:
- Check for errors daily
- Address errors immediately
- Don't let errors pile up
- Fix root causes, not just symptoms

**Action if Violated**:
- Remind user to report errors
- Check why error was ignored
- Fix error immediately
- Document error and fix

---

### 5. ❌ NEVER Allow Sharing Passwords

**Rule**: Each user must have own account, passwords must not be shared.

**Why**:
- Security risk
- Cannot track who did what
- Breaks audit trail
- Violates data discipline

**Enforcement**:
- Create account for each user
- Remind users not to share passwords
- Check for shared account usage
- Enforce individual accounts

**Action if Violated**:
- Create separate account for user
- Remind user not to share password
- Change password if shared
- Document violation

---

### 6. ❌ NEVER Allow Modifying Finalized Data

**Rule**: Finalized sales, completed production steps, etc. must not be modified.

**Why**:
- Finalized data is locked for integrity
- Modifying breaks audit trail
- Modifying affects accounting
- Modifying creates confusion

**Enforcement**:
- Lock finalized data (system should prevent)
- Remind users data cannot be modified
- Use proper correction process if needed
- Document all corrections

**Action if Violated**:
- Verify data was modified
- Check impact of modification
- Use proper correction process
- Document correction

---

### 7. ❌ NEVER Allow Bypassing Branch Selection

**Rule**: Users must select correct branch before creating data.

**Why**:
- Wrong branch = wrong inventory
- Wrong branch = wrong reports
- Wrong branch = data confusion
- Branch selection is critical

**Enforcement**:
- Remind users to check branch
- Verify branch selection in data
- Check for wrong branch data
- Correct wrong branch data immediately

**Action if Violated**:
- Remind user to select correct branch
- Correct wrong branch data
- Verify correction
- Document violation
